# Holdfast v0.2.0 Release Plan

## Goal

Ship four changes that move holdfast from "works in a demo contract directory" to "works in real projects with real async pipelines":

1. **Async `@track`** — support async pipeline functions (S)
2. **Trust model docs** — document unsandboxed custom script execution (S)
3. **Evolvable refs to source files** — point at Python symbols/YAML keys in existing source files instead of requiring extracted files in the contract dir (M-L, the headline feature)
4. **README evolution paths** — restructure docs to frame Claude Code as default path, programmatic as the automation path (S)

Items 1, 2, 4 are small. Item 3 is the real work and the reason for a minor version bump.

## Assumptions

- Python >=3.12 (can use `ast` improvements, `asyncio` patterns freely)
- No new dependencies — `ast`, `asyncio`, `functools` are all stdlib
- Evolvable refs spec exists at the cyber-genai project (referenced in memory) but we'll design fresh based on current holdfast code
- Backwards compatibility required: existing string-ref contract.yaml format must keep working
- No storage backend changes in this release (parked post-v0.2)

## Task Breakdown

### Task 1: Async `@track` decorator (Effort: S, ~30 lines)

**Current state:** `evidence.py:65-115` — synchronous wrapper only. Using `@track` on an async function would silently break (returns coroutine object instead of awaiting it).

**Approach:** Detect async functions with `asyncio.iscoroutinefunction()` and provide an async wrapper alongside the sync one. Same pattern as stdlib's `functools.wraps`.

```
# Pseudocode — not implementation
if asyncio.iscoroutinefunction(fn):
    async def wrapper(*args, **kwargs):
        # same try/except/validate/log_run logic, but with await
        result = await fn(*args, **kwargs)
        ...
    return wrapper
else:
    # existing sync wrapper unchanged
```

**Files touched:** `src/holdfast/evidence.py`, `tests/test_evidence.py`
**Risk:** Low. Additive change, sync path unchanged.
**Testing:** New test with an async function decorated with `@track`, assert evidence is logged correctly.

---

### Task 2: Trust model docs for custom scripts (Effort: S, ~10 lines of docs)

**Current state:** `validate.py:188-194` runs `subprocess.run([sys.executable, script_path], ...)` with no sandboxing. Only documented as "passes output as JSON on stdin, checks exit code" in README line 180.

**Approach:** Add a "Security considerations" note in README under the invariant types section. Call out:
- Custom scripts execute as the current user with full filesystem access
- 30s timeout is the only guard
- Fine for local dev/CI; review scripts before trusting third-party contracts

**Files touched:** `README.md`
**Risk:** None. Docs only.

---

### Task 3: Evolvable refs to source files (Effort: M-L, ~150-200 lines)

This is the headline feature. Currently, `evolvable:` entries in contract.yaml must point to files inside the contract directory. Real projects have prompts embedded in Python constants or YAML config keys. This was the blocker in the cyber-genai pipeline.

**Design options:**

#### Option A: New extractor module with ast.parse()

Add a new `src/holdfast/extract.py` module. Evolvable refs in contract.yaml gain an expanded dict format:

```yaml
evolvable:
  # Old format still works:
  prompt: "evolvable/prompt.md"

  # New: point at a Python symbol
  system_prompt:
    path: "src/pipeline/prompts.py"
    symbol: "SYSTEM_PROMPT"       # top-level assignment

  # New: point at a YAML key
  template:
    path: "config/templates.yaml"
    key: "pipelines.risk.template"  # dot-separated path
```

**Extract:** `ast.parse()` for Python (find assignment, return value). `yaml.safe_load()` + key traversal for YAML/JSON.

**Write-back:** For Python, use `ast` to locate the node, replace the string literal in the source text (preserving surrounding code). For YAML, use `ruamel.yaml` (preserves comments/formatting) or simple key-path write.

**Tradeoffs:**
- (+) Clean separation, testable extractor module
- (+) `ast.parse()` is safe — no code execution
- (+) Backwards compatible — string refs unchanged
- (-) Python write-back is fiddly (preserving formatting, handling multiline strings, f-strings)
- (-) YAML write-back needs `ruamel.yaml` for comment preservation (new dependency) or accepts losing comments

**Risk:** Medium. Python AST write-back for multiline strings and various quoting styles is the hard part.

#### Option B: Extract-only, copy-on-evolve

Same extraction logic, but when applying an evolution, write the new content to a standalone file in the contract dir (like today) and log a note saying "update source file manually." Evolution proposals never modify source files directly.

**Tradeoffs:**
- (+) Much simpler — no write-back logic
- (+) Zero risk of corrupting source files
- (-) Breaks the automated evolution loop — human must manually copy changes back
- (-) Defeats the purpose if the goal is end-to-end automation

#### Option C: Extract + simple write-back (Python only, YAML later)

Start with Python symbol extraction and write-back only. YAML/JSON support comes in a follow-up. Keeps scope tighter for v0.2.

**Tradeoffs:**
- (+) Smallest useful slice of Option A
- (+) Python is the harder problem — solving it first de-risks YAML later
- (-) Users with YAML-embedded prompts wait for v0.3

**Recommended: Option C.** Python-only evolvable refs. Covers the primary use case (prompts in Python constants), defers YAML to v0.3. Write-back uses `ast` node locations to splice new content into the source text — no `ruamel.yaml` dependency needed.

**Files touched:**
- New: `src/holdfast/extract.py` (extract + write-back)
- Modified: `src/holdfast/contract.py` (load expanded ref format, route through extractor)
- Modified: `src/holdfast/evolve.py` (pass source-file content to LLM prompt)
- Modified: `src/holdfast/version.py` (snapshot source files, apply changes via extractor)
- New: `tests/test_extract.py`
- Modified: `tests/test_integration.py` (add source-ref lifecycle test)

**Key design decisions:**
1. How to handle multiline strings (triple-quoted) — `ast` gives us line/col offsets, we can splice
2. **Decided:** Support module-level assignments AND class-attribute assignments (class-as-namespace pattern). Symbol format: `"PROMPT"` for module-level, `"ClassName.PROMPT"` for class attribute. One dot max, class must be top-level. No deeper nesting, no methods, no inner classes.
3. How snapshots work for source files (snapshot the whole file? just the symbol value?)

### Task 4: README evolution paths (Effort: S, ~30 lines changed)

**Current state:** README presents "two ways" (Python pipelines vs Claude Code only) as equal peers.

**Approach:** Restructure as a progression:
1. **Setup** — install, create contracts, wire `log_run()` / `@track`
2. **Monitor** — `check_contract()` / `python -m holdfast status`
3. **Evolve interactively** — Claude Code (recommended for monitor and semi-auto modes)
4. **Evolve programmatically** — `propose_evolution()` for auto mode / CI pipelines

**Files touched:** `README.md`
**Risk:** None. Docs only.

---

## Implementation Order

```
1. Async @track          (no dependencies, standalone)
2. Trust model docs      (no dependencies, standalone)
3. Evolvable refs        (biggest piece, do after 1-2 are merged)
   3a. extract.py — extraction logic + tests
   3b. contract.py — load expanded ref format
   3c. evolve.py — build prompts from source refs
   3d. version.py — snapshot/apply through extractor
   3e. integration test — full lifecycle with source-file ref
4. README restructure    (do last, after code is final)
```

Tasks 1 and 2 can be done in parallel. Task 3 is sequential (3a → 3b → 3c → 3d → 3e). Task 4 is last.

## Testing Strategy

| Task | Test approach |
|------|--------------|
| Async @track | Unit test: async function with `@track`, verify evidence logged. Use `pytest-asyncio` or `asyncio.run()`. |
| Trust model docs | Manual review only |
| Evolvable refs | Unit: extract symbol from .py file, write back modified value. Integration: full lifecycle with source-file ref (load → log → propose → apply → verify source file changed) |
| README | Manual review only |

## Rollback Strategy

- Each task is independently shippable — if evolvable refs slips, ship v0.2 with just async @track + docs
- Evolvable refs is backwards compatible — old contract.yaml format unchanged
- Version bump to 0.2.0 happens in the final commit

## Improvement Radar

1. [correctness] Evolvable ref write-back should round-trip without changing unrelated lines — needs careful AST offset math (Effort: baked into Task 3)
2. [tests] Property-based test for extract/write-back with random Python string literals — catches edge cases in quoting (Effort: S)
3. [clarity] Contract.yaml schema validation — currently no validation of the YAML structure itself, expanded ref format makes this more important (Effort: M)
4. [performance] `list_evidence()` loads all JSON files eagerly — not a v0.2 concern but will compound with evolvable refs adding more I/O (Effort: M, parked with storage strategy)
5. [security] Evolvable refs pointing outside contract root — should `extract.py` enforce path boundaries? (Effort: S)
